# 기술적 소통에 대한 저의 생각

저는 기술적인 토론을 즐깁니다.

상대방이 비록 신입 개발자라 할지라도, 서로 근거와 출처를 주고받다 보면 모르는 부분을 알게 되어 좋습니다.

하지만 모든 사람이 집요한 토론을 좋아하지는 않을 수 있고, 누군가는 그저 내 말을 따라주기를 바랄 수도 있습니다.

그럴 때는 토론보다 그 사람과의 원만한 관계 유지를 우선시합니다.

---

# 저의 기술적인 토론 자세

## 근거와 사실 기반의 주장을 하려 노력합니다.

### 1. 의견을 제시할 때
* **주장:** RTL의 철학에 따라, 내부 구현 상세가 아닌 **사용자가 이용하는 겉모습** 위주로 테스트를 작성해야 합니다.
* **근거:** RTL 공식 문서에서도 내부 구현 테스트를 지양하고 사용자 관점의 테스트를 권장하고 있습니다.

### 2. 반론을 수용할 때
* **반론:** "대부분의 비즈니스 컴포넌트는 그 방식에 동의합니다. 하지만 **디자인 시스템**은 예외로 두어야 합니다."
* **근거:** "디자인 시스템의 사용자는 일반 유저가 아닌 '개발자'입니다. 예를 들어 `Button`에 `primary` 타입을 주었을 때 브랜드 컬러가 정확히 적용되는지 검증하는 것은 스펙 준수 여부를 판단하는 핵심입니다."
* **신뢰할 수 있는 출처:** 실제로 **Material UI**의 테스트 코드도 스타일 속성을 검증하고 있습니다.

## 구체적으로 답변하려 노력합니다.

### 지양하는 답변
> **질문:** DTO를 왜 도입해야 하나요?

> **답변:** 확장성과 유연성이 좋아지니까요. 제 친구네 회사(카카오 등)에서도 다 그렇게 쓴대요.

> **재질문:** 구체적으로 어떤 경우에 확장성이 좋아지나요?

> **답변:** (이 당연한 걸 왜 묻냐는 표정으로) 아니, 이거 하면 확장성 좋아지잖아요;;

→ 이렇게 답변하는 동료를 좋아하는 분은 없을 겁니다.

### 제가 추구하는 답변
> API 응답 명세가 변경될 때마다 이를 사용하는 수십 개의 컴포넌트를 모두 수정해야 합니다.

> API 응답을 클라이언트용 데이터로 변환하는 DTO 레이어를 두게 되면, 명세가 바뀌어도 **변환 레이어 1곳만 수정하면 되므로 영향 범위가 획기적으로 줄어듭니다.**

이런 식으로 '왜?'를 묻고 답하며 스스로를 먼저 설득해야, 타인도 설득할 수 있다고 생각합니다.

## 요약

제가 현재 가진 모든 생각은, 이걸 반박할 수 있는 **더 타당한 논리와 근거**가 아직 없기 때문입니다.

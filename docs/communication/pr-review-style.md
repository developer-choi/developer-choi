# 코드 리뷰 피드백 및 개선 경험

> 동료하게 "집요하다" 라는 피드백을 받은 적이 있었습니다.

## 배경
저는 가능한 한 많은 영역에서 기술적 최선을 추구하는 편입니다.

> Best Practice를 쌓아나가는 것이 완성도와 시간을 모두 확보하는 길이라고 생각했었습니다.

## 피드백
코드 리뷰 할 때 공식 문서와 대기업 기술 블로그를 근거로 제 의견을 제시했으나, 동료에게 "집요하다"라는 피드백을 받았습니다.

시간적 압박이나 외부 제약이 없는 상황이다 보니, 단순한 문제 해결을 넘어 **기술적으로 가장 좋은 방법인가**를 검증하는 데 매몰되었습니다.

## 개선 방안

### 기준 1. 중요도
* **필수:** 오류 등 제품에 치명적인 경우, 기존처럼 근거를 바탕으로 설득합니다.
* **제안:** 개선 사항의 경우, "제안이며 필수는 아닙니다", "반영하지 않아도 좋습니다"라는 멘트를 명시하여 동료를 존중합니다.

### 기준 2. 상대방의 성향
* **기술적 토론을 선호하는 경우:** 토론을 통해 최적의 해답을 찾아가며 배웁니다. [(링크)](./technical-decision.md)
* **실용성을 중시하는 경우:** 문제가 없다면 동료의 구현 방식을 존중합니다.

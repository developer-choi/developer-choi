# 좋은 PR과 Commit을 작성하는 방법

## 핵심 원칙

> **작은 단위의 PR과 커밋을 지향합니다.**

이 원칙을 준수하는 것만으로도 협업 과정에서 많은 이점을 얻을 수 있습니다.

## 장점
### 1. 가독성 향상

한 번에 인지해야 할 코드의 양은 적을수록 좋습니다.

PR과 커밋 하나에 수정되거나 추가된 코드가 너무 많다면, 리뷰어가 이해하기 어렵습니다.

### 2. AI 기반 코드 리뷰 효율화

AI에게 작업을 작은 단위로 나누어 요청하고 커밋하게 되면,

AI가 생성한 결과물을 사람이 더 정확하게 검토할 수 있습니다.

### 3. 사전 논의 및 검증 역할

핵심 모듈을 모두 개발한 후 PR 리뷰 단계에서 반려되어 처음부터 다시 수정해야 하는 상황은 큰 리소스 낭비입니다.

작은 PR 원칙을 지키면 이러한 문제를 자연스럽게 예방할 수 있습니다.

#### 예시

서비스 전체에 적용할 WYSIWYG 에디터 라이브러리를 선정하고 개발해야 하는 경우:

1. 별도의 의사코드를 작성하거나,
2. 사전에 동료들을 소집하여 회의 시간을 할애하는 대신,
3. 라이브러리를 간단히 설치 및 설정하여 PR을 생성하고,
4. PR 본문에 해당 라이브러리 선정 근거를 명시하여 리뷰를 요청할 수 있습니다.

이는 실제로 동작하는 코드를 통해 방향성을 미리 공유하는 방식이므로 모호함이 줄어듭니다.

실행 가능한 코드가 더 구체적이고 명확하기 때문입니다.

### 4. 히스토리 자산화

신규 입사자를 위해 프로젝트 히스토리를 별도의 위키 문서로 정리한다면, 관리해야 할 문서의 양이 늘어나 유지보수 부담이 가중됩니다.

잘 작성된 PR 링크를 첨부함으로써 이러한 히스토리 문서를 효과적으로 대체할 수 있습니다.

## 결론

문서, 커밋 메시지, 소스 코드 등 모든 리소스는 각자의 최적화된 위치가 있습니다.

PR과 커밋 메시지가 본연의 역할을 충실히 수행한다면, 별도의 복잡한 프로세스 없이도 많은 문제를 해결할 수 있습니다.

가장 중요한 것은 '기본'을 지키는 것입니다.

## PR 본문에는 무엇을 작성하나요?

1. **문제 정의 (목표)**: 무엇을 해결하고자 하는가?
2. **해결 방안 탐색**: 고려했던 다양한 방법들
3. **의사 결정 및 근거**: 선택한 방법과 그 이유
4. **참고 자료**: 기획서, 디자인 시안 등 관련 링크

**구현된 코드에 대한 설명은 최소화하는 것을 목표로 합니다.**

코드에 대한 설명이 필요하다는 것은, 역설적으로 코드 자체의 가독성이나 구조에 개선이 필요하다는 신호일 수 있습니다.

### 코드 설명을 대체하는 방법

- **1순위**: 코드 자체의 가독성을 높여 설명이 필요 없게 만듦
- **2순위**: 커밋 메시지를 통해 문맥과 의도를 전달

#### 코드 자체로 설명을 대체하는 예시

**Before:**
```javascript
if (activeElement === sentinelStart.current || activeElement === sentinelEnd.current) {  
 tabbable = defaultGetTabbable(rootRef.current!);  
}
```

**After:**
```javascript
const isFocusAtBoundary = activeElement === sentinelStart.current || activeElement === sentinelEnd.current;  
if (isFocusAtBoundary) {  
 tabbable = defaultGetTabbable(rootRef.current!);  
}
```

복잡한 조건문을 의미 있는 변수명(`isFocusAtBoundary`)으로 추상화함으로써, "이 조건이 무엇을 의미하는가?"에 대한 답을 코드 자체가 제공하도록 개선할 수 있습니다.

#### 커밋 메시지로 설명을 대체하는 예시

**코드 내 주석:**
```typescript
function publishBanner(banner: Banner) {  
 // 운영팀 요청으로 이 유저는 배너를 게시하지 못하게 수정
 if (user.id === 10001) {  
   return;  
 }  
}
```

이러한 주석 대신, 해당 내용을 커밋 메시지에 포함시킵니다.

동료들은 커밋 로그와 PR 히스토리를 통해 "왜 이 코드가 추가되었는지"에 대한 배경을 명확히 파악할 수 있습니다.

## 커밋은 어떤 단위로 작성하나요?

커밋 하나에 하나의 주제만 담기도록 **Squash** 전략을 적극적으로 활용합니다.

```text
fix: 이것도 한계가 있어서 다시 A 기술로 변경
fix: B 기술이 더 좋아보여서 B 기술로 변경
feat: 상품 카드 컴포넌트를 A 기술로 구현
```

단순 실수나 중간 과정은 히스토리로서의 가치가 낮으므로, 최종적으로 하나의 의미 있는 커밋으로 합치는 것이 좋습니다.

하지만 과정 자체가 중요한 히스토리가 되는 경우도 있습니다.

### 커밋을 분리하여 남기는 기준 (히스토리 보존)
- 기획이나 디자인 스펙이 변경된 경우
- 동료와의 협의를 통해 기술적 의사결정이 변경된 경우

## 커밋 작성 순서

큰 규모의 작업을 진행할 때는 커밋을 논리적인 단위로 잘게 나누어 진행해야 충돌을 방지하고 리뷰 효율을 높일 수 있습니다.

예시: 새로운 디자인과 API를 적용한 유저 정보 페이지 구현

### Step 1. Deprecated 처리
기존 V1 API를 사용하는 모든 모듈에 `@deprecated` 태그를 작성합니다.

### Step 2. 기존 코드 리팩토링 및 구조 개선
V1과 V2 API 코드가 공존하는 과도기적 상황을 대비하여, 파일 위치를 재정비하고 재사용 가능한 코드를 추상화합니다.

### Step 3. V2 API 타입 정의
새로운 API의 Request/Response 타입과 도메인 모델 타입을 사전에 정의합니다. 이는 후속 커밋의 기반이 됩니다.

### Step 4. API 호출 레이어 구현
V2 API를 호출하고, 응답 데이터를 프론트엔드 도메인 모델에 맞게 변환하는 함수를 작성합니다.

이 단계에서 에러 핸들링 로직을 포함하여 구체적인 도메인 에러를 반환하도록 처리합니다.

### Step 5. 단위 컴포넌트 및 유틸리티 개발
페이지를 구성하는 최소 단위의 컴포넌트(Button, Card 등)나 유틸리티 함수를 개발하고 커밋합니다.

이 과정에서 단위 테스트를 함께 작성합니다.

### Step 6. 컴포넌트 조립
단위 컴포넌트들을 조합하여 더 큰 단위의 위젯이나 페이지 레이아웃을 구성합니다.

필요 시 통합 테스트를 작성합니다.

### Step 7. 실제 데이터 연동 및 상태 처리
API 호출 로직을 컴포넌트와 연결하고, 비동기 상태(Loading, Success, Failure) 및 엣지 케이스를 처리합니다.

1. **Loading**: 스켈레톤 UI 또는 로딩 인디케이터 적용
2. **Success**: 데이터 렌더링
3. **Failure**: 4xx 5xx 에러 등 예외 상황에 대한 사용자 피드백
4. **Edge Case**: 목록이 없는 경우 처럼 성공했지만 기대한 상황은 아닌 경우

이 단계에서 E2E 테스트를 추가할 수 있습니다.

### Step 8. QA 및 피드백 반영
QA 과정에서 발견된 버그나 기획 변경 사항을 별도의 커밋으로 반영합니다.
예: `feat(user): 기획 변경으로 인한 xxx 로직 수정`

# AI 시대에 저는 이렇게 성장하고 있습니다

AI 시대 개발자의 가치는 두 가지라고 생각합니다. **일을 만들어서 AI에게 시키는 것**, 그리고 **AI가 만든 결과물을 잘 리뷰하는 것**.

---

## 1. 잘 시키기

AI에게 잘 시키려면 두 가지가 필요합니다.

- **Best Practice** — 무엇을 만들지 기준이 됩니다.
- **스킬** — 한 번 내린 판단을 프롬프트로 만들어 AI에게 위임합니다.

### 1-1. Best Practice를 쌓아 AI가 복제하게

한 번 정립한 best practice는 AI로 빠르게 복제할 수 있습니다. 속도와 완성도를 동시에 가져갈 수 있는 가장 좋은 방법이라고 생각합니다.

- **Sentry 구축** — 에러 모니터링 도입 및 이슈 98% 해결
- **에러 처리 로드맵** — 예방부터 대응까지 7단계로 정리

#### Sentry 구축 및 이슈 98% 해결

에러가 발생했을 때 빠르게 대응하려면 모니터링이 필요합니다. Sentry를 도입하여 프론트엔드 에러를 실시간으로 수집하고, 발견된 이슈의 98%를 해결했습니다.

자세한 구현은 [PR #3](https://github.com/developer-choi/best-practice/pull/3)을 참고해 주시면 감사하겠습니다.

#### 프론트엔드 에러 처리 로드맵: 7 Steps

에러 처리보다 더 중요한 건 애초에 에러가 발생하지 않도록 예방하는 것입니다. 그럼에도 발생했다면 사용자에게 잘 안내하고 빨리 대응해야 합니다.

이 두 가지 목표를 달성하기 위해 에러 피드백 UX 설계, 에러 처리 원칙, CSR/SSR/이벤트 핸들러 시점별 처리, 전역/공통/개별 처리까지 7단계로 정리했습니다.

자세한 내용은 [에러 처리는 왜 필요한가?](https://github.com/developer-choi/developer-choi/blob/main/docs/error-handling/step1.md)를 참고해 주시면 감사하겠습니다.

### 1-2. 프롬프트로 일하는 방식을 위임

한 번 내린 판단을 AI에게 위임하여, 같은 결정을 두 번 하지 않게 합니다. 두 가지 방식으로 활용하고 있습니다.

- **스킬** — 한 번 내린 판단을 프롬프트로 만들어 AI에게 위임합니다.
- **AI가 먼저 제안** — 방향을 정해두면 AI가 사람이 놓친 것을 발굴합니다.

#### Workflow 스킬

제가 일하는 방식 자체를 프롬프트로 옮겼습니다. 기획서나 피그마를 입력하면 PR이 나옵니다.

> - Step 1 **배경 파악**: 주어진 자료를 읽고 무엇을 왜 해야 하는지 정리
> - Step 2 **PR 분할**: 리뷰하기 좋은 크기로 작업을 나눔
> - Step 3 **과제 정의**: 각 PR의 목표와 접근 방식 결정
> - Step 4 **구현 방침**: 구체적인 구현 계획 수립
> - Step 5 **구현**: 코드 작성 및 커밋
> - Step 6 **최종 점검**: 코드 품질과 설계 일관성 검증
> - Step 7 **PR 본문 작성**: 리뷰어를 위한 PR 설명 작성
> - Step 8 **마무리**: 회고 및 제출 절차 마무리

자세한 내용은 [workflow 스킬](https://github.com/developer-choi/ai-contexts/blob/master/deploy/skills/workflow/README.md)을 참고해 주시면 감사하겠습니다.

#### Spec Review 스킬

미리 방향을 정해두면, AI가 그 안에서 사람이 미처 보지 못한 것을 발굴합니다.

기획서나 피그마를 받으면 코드를 짜기 전에 개발자 관점에서 빠진 부분이 없는지 점검합니다.

> - 관심사별 점검: 화면 목적, 사용자 동선, 에러/예외, 데이터, 라우팅, 디자인
> - 페이지 특성별 점검: 폼, 목록, 순차 플로우, 상세 페이지
> - 완성도(404, 파비콘, OG 태그), UX(뒤로가기, 빈 상태, 토스트 피드백) 등 사람이 미처 챙기지 못하는 디테일까지 TODO로 쌓아둡니다
> - 사용자와 AI가 서로 아이디어를 주고받으며 구현 스펙을 만들어갑니다

자세한 내용은 [spec-review 스킬](https://github.com/developer-choi/ai-contexts/blob/master/deploy/skills/workflow/spec-review/README.md)을 참고해 주시면 감사하겠습니다.

## 2. 잘 리뷰하기

잘 리뷰하기 위한 방법은 두 가지입니다.

- **전문성을 쌓기** — 공식 문서를 기반으로 학습합니다.
- **리뷰 비용을 줄이는 구조** — 사람이 리뷰할 양 자체를 줄입니다.

### 2-1. 전문성을 쌓기

AI가 작성한 코드를 리뷰하려면 전문성이 필요합니다. 이를 위해 공식 문서를 기반으로 학습하고 있으며, 영어 원문을 기록하고 기술 면접 형식으로 복습합니다.

공식 문서의 영어 원문을 읽고 Q&A 형태로 기록합니다. 축적된 Q&A를 기반으로 시니어 개발자가 기술 면접을 진행하는 것처럼 복습합니다. 답을 알려주지 않고, 빠진 부분은 꼬리 질문으로 유도합니다.

### 2-2. 리뷰 비용을 줄이는 구조

AI가 작성하는 코드의 양이 늘수록, 사람이 리뷰해야 할 양도 늘어납니다. 이를 해결하는 방법은 리뷰 양을 줄이는 것입니다.

비용이 저렴한 순서로 정리하면:

- **정적 분석** — 린트가 커밋 시점에 실수를 자동 차단
- **테스트 코드** — 기존 동작이 깨지지 않았는지 자동 확인
- **AI 코드리뷰** — 린트가 잡지 못하는 문제를 AI가 담당

#### 2-2-1. 정적 분석

```typescript
const first: string = users[5];  // 실제로는 undefined — 타입은 정상으로 통과
```

이런 코드는 동작은 하지만 버그를 품고 있습니다. 사람이 수백 줄을 리뷰하면서 이런 패턴을 매번 찾아내기는 어렵습니다. 린트가 커밋 시점에 자동으로 차단하면, 리뷰어는 로직과 설계에만 집중할 수 있습니다.

19개 이상의 커스텀 린트 룰을 적용했습니다 — [PR #11](https://github.com/developer-choi/monorepo-playground/pull/11)

#### 2-2-2. 테스트 코드

```typescript
expect(screen.getByRole("img", { name: /배너/ })).toBeInTheDocument();
```

AI가 코드를 생성하거나 수정할 때마다, 기존 동작이 깨지지 않았는지 테스트가 자동으로 확인합니다. 이를 통해 사람이 검증해야 할 범위가 줄어듭니다 — [PR #4](https://github.com/developer-choi/best-practice/pull/4)

#### 2-2-3. AI 코드리뷰

린트가 잡지 못하는 문제는 AI 코드리뷰가 담당합니다. 반복된 지적은 컨벤션에 축적됩니다. 이를 통해 사람이 리뷰에 쓰는 시간이 줄어듭니다.

자세한 내용은 [code-review 스킬](https://github.com/developer-choi/ai-contexts/blob/master/deploy/skills/workflow/code-review/README.md)을 참고해 주시면 감사하겠습니다.

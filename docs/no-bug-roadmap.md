# 버그 없는 프로그램 & AI가 작성한 코드를 잘 리뷰해서, 생산성 올리기

## 준비할 서사

### 문제

1. 코드리뷰(자체/타인) 및 QA(자체 테스트/타인 테스트)에서 발견되는 오류 건수를 종류별 n건

### 해결방법

자동화된 안전망을 비용이 낮은 순서대로 도입한다:

1. **Static Analysis** (린트, 타입 체크)
   - ESLint 룰과 tsconfig 옵션을 전수조사하여, 버그 방지 룰과 컨벤션 룰 모두 적용한다.
   - 근거: 사람은 회사 컨벤션과 도메인 지식이라는 암묵지가 있어 실수 범위가 좁지만, AI는 예측 불가능하게 잘못 쓸 수 있으므로 안전망을 넓게 깔아야 한다.
   - 컨벤션 위반도 사람 코드리뷰에서 지적이 발생하는 항목이므로, 자동화 대상에 포함한다.
2. **테스트** (자동화된 테스트 코드)
   - 테스트 종류(E2E/통합/유닛)와 비율은 일반적인 테스트 피라미드 이론이 아니라, 측정 결과에서 나온 오류 유형에 의해 결정한다.
   - 꼬리질문 대비: "AI가 만드는 오류를 잡기 위해 구체적으로 어떤 테스트를 작성했는가?" → 측정된 오류 유형별로 해당 유형을 방지하는 테스트를 작성했다고 답변해야 한다.
3. **AI 코드리뷰**
   - 린트와 테스트로 잡지 못하는 오류 중, 한 번 이상 발생했거나 사람이 PR에서 지적한 패턴을 AI 리뷰 프롬프트/체크리스트로 관리하여 재발을 방지한다.
   - 오류 수집 자동화: 타인 코드리뷰 지적은 `gh api`로 PR 코멘트를 일괄 수집하고, QA 오류 및 자체 코드리뷰 지적은 수동 수집한다.
   - 분류 자동화: 수집된 코멘트를 AI가 "린트로 해결 가능 / 테스트로 해결 가능 / AI 리뷰로 해결 가능 / 자동화 불가"로 분류하고, 사람이 리뷰하여 확정한다.
   - 확정된 분류에 따라 각 안전망(린트 룰 추가, 테스트 작성, AI 리뷰 체크리스트 갱신)에 반영한다.

이 안전망들의 목표는 수동 프로세스(사람 코드리뷰, 사람 테스트, QA 테스트)에서 발견되는 오류 건수를 줄이는 것이다.

### 선택 근거

- 전부 도입해야 하는 상황에서, 비용이 저렴한 것부터 적용하면 즉시 개선이 시작되고 그 사이 비용이 큰 것을 준비할 수 있다.
- 우선순위는 측정 결과에서 가장 많이 나온 유형과 자동화 가능 여부를 함께 고려하여 결정한다.

### 구현 과정

#### 1회 발견 즉시 자동화하여 재발 방지

자동화 가능한 지적은 **1회 발생 시 즉시**, 가장 저렴한 수단(Static Analysis → 테스트 → AI 코드리뷰 순)으로 자동화한다. 자동화 불가능한 지적(맥락 판단, 변수명 등)은 이 원칙의 대상이 아니다.

#### 안전망 실행 시점

| 시점 | 도구 | 검사 범위 |
|------|------|-----------|
| 커밋 전 (pre-commit) | husky + lint-staged | 변경된 파일만 |
| PR 생성 시 | GitHub Actions | 전체 프로젝트 |

- 커밋 시에는 lint-staged로 변경 파일만 검사하여 프로젝트 규모와 무관하게 속도를 유지한다.
- PR에서 전체 검사를 돌려 커밋 시 누락된 것을 최종 방어한다.
- git hooks를 우회하는 경우를 대비하여 PR의 GitHub Actions가 최후의 안전망 역할을 한다.

오류 건수를 종류별로 N건 → M건으로 감소시킨 수치를 제시한다. (문제와 동일한 지표)

## 실무 도입 전략

### 1단계: 측정 (도입 전 baseline 확보)

- 코드리뷰/QA에서 발생한 오류·지적을 종류별로 기록·분류한다.
- 타인 코드리뷰 지적은 `gh api`로 PR 코멘트를 일괄 수집한다.
- 이 데이터가 이력서 서사의 "문제" 수치이자 "성과" 비교 기준선이 된다.

### 2단계: 동료 설득 및 린트 도입

- 기존 린트 설정 오류 복구를 먼저 진행한다. (반대 명분 없음)
- 복구하는 김에 ESLint/tsconfig 전수조사 후, 필요한 룰을 추가 제안한다.
- 각 룰마다 before & after 예시를 준비하여 "이 룰이 없으면 이런 코드가 통과된다 / 있으면 잡힌다"를 보여준다.
- PR 코멘트에서 실제 지적된 사례가 있는 룰은 해당 사례를 근거로 함께 제시한다.

### 3단계: 측정 (도입 후 성과 확인)

- 1단계와 동일한 기준으로 오류·지적 건수를 측정하여 도입 전후를 비교한다.

## 무한스크롤 게시글 목록 (PR 4개 시리즈)

### 의도

게시글 목록 한 페이지를 처음부터 끝까지 직접 구현하며, 실서비스 수준의 성능·안정성을 갖추기까지 어떤 결정이 필요한지를 단계적으로 학습. 단순 동작이 아니라 첫 응답 속도·트래픽·렌더링 비용·실패 안전성까지 4 PR에 걸쳐 검증.

### 개선 과정

- 렌더링 전략: CSR → SSR Prefetch → Streaming 단계적 적용. API 1초 지연 기준 서버 첫 응답 1.04s → 0.01s
- 이미지 최적화: JS 런타임 / next/image / srcset 비교 후 srcset + Cloudinary 리사이징 채택. 773kB → 20.2kB (97% 감소)
- 스크롤 감지: scroll 이벤트의 throttle ↔ 정확도 트레이드오프를 IntersectionObserver로 해소. 콜백 32회+ → 1회
- 캐싱 전략: 무한 쿼리는 staleTime 만료 시 누적된 모든 페이지가 동시에 리페칭되는 동작(window focus 포함) 발견 → staleTime: Infinity + gcTime: 30s로 차단
- 리렌더링 최적화: 새 페이지 로드 시 기존 카드 전부 리렌더링 → React.memo로 5페이지 기준 600회 → 120회 (80% 감소)
- 에러 방어: 다음 페이지 API 실패 시 무한 재요청 발생 → 다음 페이지 요청 활성 조건에 에러 상태를 포함해 차단, 빈 목록 안내 추가

### 배운 점

- 직관적인 첫 선택(CSR / 원본 이미지 / scroll 이벤트 / 기본 staleTime / memo 없음 / 실패 무방어)은 대부분 한계가 있고, 그 한계는 개발자도구의 네트워크·렌더 카운트로 직접 측정해야 보임
- TanStack Query 같은 고수준 라이브러리도 누적 페이지처럼 특수한 케이스에선 기본값이 함정이 됨 — 라이브러리 동작을 한 번 더 의심하는 습관
- 성능 최적화는 한 번에 결정되지 않고, "측정 → 가설 → 대안 비교 → 채택" 사이클을 반복해야 답이 나옴

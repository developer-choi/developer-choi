# 자동 배포 시스템 구축

6단계는 2단계에서 수행했던 수동 배포 과정을 완전히 자동화하여, 휴먼 에러를 방지하고 팀 전체의 릴리스 속도를 극대화하는 최종 단계입니다.

---

## 문제 정의

* **수동 배포의 위험성**: 여러 패키지의 버전을 수동으로 올리고 배포하는 과정에서 특정 패키지를 누락하거나 잘못된 태그를 붙이는 실수가 발생함.
* **변경 이력 관리 부재**: 어떤 패키지가 어떤 이유로 업데이트되었는지에 대한 기록(CHANGELOG)이 수동으로는 누락되기 쉽고 파편화됨.
* **배포 병목 현상**: 특정 인원(관리자)만 배포 권한을 가지거나 배포 방식을 알고 있어, 릴리스가 필요한 시점에 즉각적인 대응이 어려움.

---

## 성과 정의

* **완전 자동화된 릴리스 파이프라인**: 메인 브랜치에 코드가 병합되면 별도의 수동 작업 없이 테스트, 버전 업, 배포가 한 번에 완료됨.
* **시맨틱 버전 자동화**: Changesets와 같은 도구를 활용하여 변경 사항에 맞는 버전(Major, Minor, Patch)을 지능적으로 계산하고 적용함.
* **배포 신뢰성 확보**: 자동화된 품질 검사(Test, Lint)를 통과한 코드만 배포되도록 강제하여 프로덕션 환경의 안정성을 보장함.

## 상세 실행 전략

### 자동 릴리스 자동화 (Changesets)

모노레포의 복잡한 버전 관계를 지능적으로 해결합니다.

* **Changesets 연동**: 개발자가 PR 단계에서 변경 사항의 성격(Patch, Minor, Major)을 미리 작성하고, 병합 시 이를 합쳐 버전 업데이트를 수행합니다.
* **버전 충돌 방지**: 의존 관계에 있는 패키지들의 버전을 터보레포와 연동하여 일괄적으로 관리합니다.

### CI/CD 파이프라인 통합 (GitHub Actions)

* **배포 워크플로우**: GitHub Actions를 통해 코드가 병합되면 자동으로 `npm publish`를 수행하며, 필요 시 사설 NPM 레지스트리에 배포하도록 구성합니다.
* **품질 게이트웨이**: 빌드 및 테스트 타스크가 실패할 경우 배포를 중단하여 결함이 있는 패키지가 외부로 나가는 것을 방지합니다.

### 변경 이력 문서화

* **CHANGELOG 자동 생성**: 각 패키지 폴더 내에 자동으로 수정 사항을 기록하여, 이 패키지를 사용하는 외부 개발자가 변경 내용을 한눈에 파악하게 합니다.

## 도입 사례: Docker 레이어 캐싱

GitHub Actions에서 매 빌드마다 Docker 이미지를 처음부터 빌드하는 대신, 원격 클라우드 캐시에 이미지 레이어를 저장하여 재사용합니다.

* **결과**: 프로젝트 빌드 시간 평균 **300초 → 210초 (30% 단축)**.
